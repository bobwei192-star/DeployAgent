# Jenkins 端口自动分配未传递给子部署脚本

## 问题现象

在服务器执行完整部署：

```bash
cd ~/DevOpsAgent/deploy
sudo python3 deploy_all.py
```

选择模式 `1`，即完整部署 `Jenkins + GitLab + MantisBT + Nginx` 后，主脚本端口扫描阶段已经识别到默认端口被占用，并自动分配了新端口：

```text
[jenkins/web] 18081 已被占用 → 自动分配 18082
[jenkins/agent] 50000 已被占用 → 自动分配 50001
```

但后续 Jenkins 子部署脚本实际创建容器时仍然绑定旧端口：

```text
Bind for 127.0.0.1:18081 failed: port is already allocated
```

最终 Jenkins 部署失败：

```text
jenkins 部署脚本返回 exit=125
jenkins 部署失败 (exit=125)
```

## 影响范围

- 影响 `deploy_all.py` 的交互式完整部署和 standalone 部署路径。
- 当默认端口被已有服务或旧容器占用时，主脚本虽然能扫描并计算新端口，但 Jenkins/GitLab/MantisBT/Nginx 子脚本仍可能读取默认端口或 `.env` 中旧端口。
- 直接影响 Jenkins：
  - `JENKINS_PORT_WEB`
  - `JENKINS_PORT_AGENT`
- 同类风险也存在于：
  - `GITLAB_PORT_HTTP`
  - `GITLAB_PORT_HTTPS`
  - `GITLAB_PORT_SSH`
  - `MANTISBT_PORT_WEB`
  - `MARIADB_PORT`
  - `NGINX_PORT_JENKINS`
  - `NGINX_PORT_GITLAB`
  - `NGINX_PORT_MANTISBT`

## 根因分析

`deploy_all.py` 中 `scan_ports()` 会返回自动分配后的 `port_map`，但原部署流程没有保存该返回值，也没有把 `port_map` 应用到当前进程环境变量中。

修复前流程等价于：

```python
scan_ports(services)
scan_docker_network()
scan_volumes()
deploy_services(services, use_nginx, bind)
```

问题点：

1. `scan_ports(services)` 的返回值被丢弃。
2. 自动分配的新端口没有写入 `os.environ`。
3. `deploy_service()` 调用 shell 子脚本时，只额外传递了卷名环境变量。
4. `deploy_jenkins.sh` 只能读取环境变量或默认值：

```bash
JENKINS_PORT_WEB="${JENKINS_PORT_WEB:-18081}"
JENKINS_PORT_AGENT="${JENKINS_PORT_AGENT:-50000}"
```

因此即使主脚本已经计算出 `18082`，子脚本仍回退到默认 `18081`。

### 二次复现后的补充根因

第一次修复后，主脚本已经能把自动分配端口写入 `os.environ` 并传给子部署脚本，但服务器二次复现仍失败：

```text
[jenkins/web] 18081 已被占用 → 自动分配 18082
端口分配已写入: /home/jenkins/DevOpsAgent/deploy/.env.auto
...
加载环境变量: /home/jenkins/DevOpsAgent/deploy/.env
...
Bind for 127.0.0.1:18081 failed: port is already allocated
```

进一步分析发现，Jenkins 子脚本会 `source lib/common.sh`，并在部署过程中调用 `load_env()` 加载 `.env`。`lib/common.sh::load_env()` 修复前会无条件执行：

```bash
export "$var_name"="$var_value"
```

这会把主脚本传入的 `JENKINS_PORT_WEB=18082` 覆盖回 `.env` 中旧值 `JENKINS_PORT_WEB=18081`。

也就是说，真实根因分两层：

1. 主脚本最初没有应用 `port_map`。
2. 主脚本应用 `port_map` 后，子脚本内部又用 `.env` 覆盖了自动端口。

### 三次复现后的补充根因：Nginx 子脚本暴露额外端口

第二次修复后，Jenkins/GitLab/MantisBT 已经可以部署成功，说明自动端口已正确传递到业务服务子脚本：

```text
✓ jenkins 部署完成
✓ gitlab 部署完成
✓ mantisbt 部署完成
```

新的失败点转移到 Nginx：

```text
创建 Nginx 容器...
  - 端口 Jenkins: 0.0.0.0:18443 -> 8440
  - 端口 GitLab: 0.0.0.0:18444 -> 8441
  - 端口 Agent: 0.0.0.0:18442 -> 8442
  - 端口 MantisBT: 0.0.0.0:18445 -> 8443
  - 端口 Registry: 0.0.0.0:18444 -> 8444
  - 端口 Harbor: 0.0.0.0:18445 -> 8445
  - 端口 SonarQube: 0.0.0.0:18446 -> 8446
```

随后 Docker 报错：

```text
Bind for 10.67.69.34:18442 failed: port is already allocated
```

根因是 `deploy_all.py` 在完整部署模式中仍把 `nginx` 当作普通服务，调用了旧的 `deploy_nginx/deploy_nginx.sh`。

该 shell 脚本不仅暴露 Jenkins/GitLab/MantisBT 三个本次需要的反向代理端口，还会固定暴露：

- `NGINX_PORT_AGENT`
- `NGINX_PORT_REGISTRY`
- `NGINX_PORT_HARBOR`
- `NGINX_PORT_SONARQUBE`

这些端口没有纳入 `deploy_all.py` 当前 `PORT_REGISTRY["nginx"]` 的自动扫描范围，且还会和自动分配后的 Jenkins/GitLab/MantisBT Nginx 端口发生内部冲突：

```text
nginx/gitlab   自动分配 18444
nginx/registry 默认使用 18444

nginx/mantisbt 自动分配 18445
nginx/harbor   默认使用 18445
```

因此完整部署模式不应该先跑旧的独立 Nginx 子脚本，而应该由 `deploy_all.py::ensure_nginx_proxy()` 根据当前实际运行的后端容器动态生成并启动 Nginx。

### 四次复现后的补充根因：Nginx 证书 key 缺失但误报启动成功

第三次修复后，部署流程显示完成，并输出了 HTTPS 地址：

```text
【Nginx HTTPS 访问地址】
  jenkins: https://10.67.69.34:18443
  gitlab: https://10.67.69.34:18444
  mantisbt: https://10.67.69.34:18445
```

但浏览器访问时报：

```text
ERR_CONNECTION_REFUSED
```

排查命令：

```bash
docker ps --filter name=devopsagent --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
docker logs --tail=120 devopsagent-nginx
ss -tlnp | egrep '18443|18444|18445|8440|8441|8443'
```

关键输出：

```text
devopsagent-nginx   Restarting (1)
cannot load certificate key "/etc/nginx/ssl/devopsagent.key":
BIO_new_file() failed ... No such file or directory
```

说明 Nginx 容器实际一直在重启，宿主机并没有监听 `18443/18444/18445`，所以浏览器连接被拒绝。

根因在 `deploy_all.py::ensure_nginx_proxy()` 的证书生成逻辑：

```python
if not crt_path.exists():
    # generate crt and key
```

该逻辑只检查 `devopsagent.crt` 是否存在。如果历史部署中残留了 `devopsagent.crt`，但 `devopsagent.key` 缺失，脚本会跳过证书生成，最终 Nginx 配置引用：

```nginx
ssl_certificate /etc/nginx/ssl/devopsagent.crt;
ssl_certificate_key /etc/nginx/ssl/devopsagent.key;
```

导致容器启动后立即退出。

此外，旧实现只检查 `docker run -d` 返回码。`docker run -d` 成功只表示容器创建成功，不代表 Nginx 主进程配置正确、容器保持 running。因此出现了：

```text
Nginx reload 返回非 0，请检查容器日志
✓ Nginx 已启动
```

这种误报成功。

### 1. 新增 `.env` 加载函数

新增 `apply_env_file()`，用于把 `.env` 中的配置加载进当前 Python 进程环境变量，保证后续子脚本可以继承。

```python
def apply_env_file():
    """Load .env values into current process environment for child scripts."""
    env_vars = load_env()
    if not env_vars:
        return
    os.environ.update(env_vars)
```

### 2. 新增端口环境变量映射函数

新增 `_set_port_env()`，把 `scan_ports()` 得到的端口映射转换成 shell 子脚本能识别的环境变量。

普通服务规则：

Nginx 特殊规则：

### 3. 新增 `apply_port_map()`

新增 `apply_port_map()`，负责两件事：

- 更新内存中的 `PORT_REGISTRY`，保证主脚本后续 summary 和 Nginx 配置使用新端口。
- 更新 `os.environ`，保证 shell 子部署脚本继承新端口。

```python
def apply_port_map(port_map):
    """Apply auto-selected ports to env and in-process registry."""
    for service_name, ports in port_map.items():
        if isinstance(ports, dict):
            if service_name in PORT_REGISTRY and isinstance(PORT_REGISTRY[service_name], dict):
                PORT_REGISTRY[service_name].update(ports)
            for port_name, port in ports.items():
                _set_port_env(service_name, port_name, port)
        else:
            PORT_REGISTRY[service_name] = ports
            os.environ[f"{service_name.upper()}_PORT"] = str(ports)
```

### 8. 跳过旧 Nginx 子脚本，统一使用 Python 内置反向代理编排

修改 `/home/zx/CICD/DevOpsAgent/deploy/deploy_all.py::deploy_services()`。

修复前：

```python
def deploy_services(services, use_nginx, nginx_bind="0.0.0.0"):
    _ensure_network()
    for svc in services:
        if not deploy_service(svc):
            return False
    if use_nginx:
        ensure_nginx_proxy(nginx_bind)
    return True
```

问题是当 `services` 包含 `nginx` 时，会先执行：

```text
deploy_nginx/deploy_nginx.sh --deploy
```

然后才执行 Python 内置的 `ensure_nginx_proxy()`。旧 shell 脚本会暴露额外端口，容易和当前部署模式无关的服务发生冲突。

修复后：

```python
def deploy_services(services, use_nginx, nginx_bind="0.0.0.0"):
    _ensure_network()
    for svc in services:
        if svc == "nginx" and use_nginx:
            info("跳过独立 Nginx 子脚本，使用 deploy_all.py 内置反向代理编排")
            continue
        if not deploy_service(svc):
            return False
    if use_nginx:
        return ensure_nginx_proxy(nginx_bind)
    return True
```

修复后的行为：

- 完整部署模式仍然选择 `nginx`，但不会调用旧 shell 子脚本。
- `deploy_all.py::ensure_nginx_proxy()` 会检测实际运行的后端容器。
- 只为当前存在的后端服务发布端口：
  - Jenkins
  - GitLab
  - MantisBT
- 不再发布本次部署不需要的 Agent/Registry/Harbor/SonarQube 端口。

### 五次复现后的补充根因：GitLab 生成 HTTP 重定向

Nginx 成功启动后，访问 GitLab 根路径：

```bash
curl -k -I https://10.67.69.34:18444/
```

返回：

```text
HTTP/1.1 302 Found
Location: http://10.67.69.34:18444/users/sign_in
```

直接访问登录页：

```bash
curl -k -I https://10.67.69.34:18444/users/sign_in
```

返回：

```text
HTTP/1.1 200 OK
```

这说明 Nginx HTTPS 本身可用，但 GitLab 后端认为自己的外部访问地址是 HTTP，所以从 `/` 重定向到：

```text
http://10.67.69.34:18444/users/sign_in
```

浏览器随后用明文 HTTP 请求 HTTPS 端口 `18444`，Nginx 因此返回：

```text
400 Bad Request
The plain HTTP request was sent to HTTPS port
```

根因是部署 GitLab 时没有在子脚本执行前强制设置 HTTPS 反向代理相关变量：

```text
GITLAB_USE_HTTPS_PROXY=true
GITLAB_HOSTNAME=10.67.69.34
GITLAB_NGINX_PORT=18444
GITLAB_EXTERNAL_URL=https://10.67.69.34:18444
```

`deploy_gitlab.sh` 已经支持这些变量，但 `deploy_all.py` 之前没有根据自动分配后的 Nginx 端口和用户确认的服务器 IP 主动设置它们。

### 12. 修复 GitLab HTTPS external_url 配置

修改 `/home/zx/CICD/DevOpsAgent/deploy/deploy_all.py`，新增：

```python
def configure_reverse_proxy_env(services, use_nginx, public_host):
    """Configure backend services before deploying behind HTTPS Nginx."""
    if not use_nginx:
        return

    nginx_ports = PORT_REGISTRY.get("nginx", {})
    if "gitlab" in services and isinstance(nginx_ports, dict):
        gitlab_port = nginx_ports.get("gitlab")
        if gitlab_port:
            os.environ["GITLAB_USE_HTTPS_PROXY"] = "true"
            os.environ["GITLAB_HOSTNAME"] = public_host
            os.environ["GITLAB_NGINX_PORT"] = str(gitlab_port)
            os.environ["GITLAB_EXTERNAL_URL"] = f"https://{public_host}:{gitlab_port}"
```

主流程在用户确认服务器 IP 后、部署服务前调用：

```python
configure_reverse_proxy_env(services, use_nginx, bind_ip)
```

这样 GitLab 容器创建时会得到正确的：

```text
external_url 'https://10.67.69.34:18444'
```

并进入 HTTPS 反向代理模式。

        log_info "加载环境变量: $env_file"
